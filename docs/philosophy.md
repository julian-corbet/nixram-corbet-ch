# Philosophy (working draft)

Julian's design arguments, kept as one running table, refined statement by
statement. Working draft — feeds `site/index.html` and `docs/rationale.md`
once settled.

| # | Axis | Rule | Split / Value |
|---|---|---|---|
| 1 | Compressed pool contents | Only cold pages may live compressed — danger is which pages end up there, not how big the pool is | — |
| 2 | zram vs zswap | Branches on mass-storage speed, checked at config time (not runtime auto-probe) | fast storage → zswap; slow/no storage → zram |
| 3 | Physical leg (RAM committed to compressed data) | Same percentage in zram and zswap; the two modes differ only in what sits behind that leg (zswap: real disk overflow; zram: none) | **Built, attribution corrected.** 30% (256M-1G) / 25% (2G-16G) / 20% (24G-128G). Only two of these are Julian's own stated figures: 30% (from the e2-micro/vultr/256M walkthrough) and 20% at ~128G ("a 20% slice... about 25GB"). The 25% band, and — separately — exactly *where* the step down to 20% begins (24G, not 32G or 64G), are Claude's own extrapolated placement filling the gap between his two data points, not something Julian specified; see `rationale.md [2]`. A later adversarial pass briefly bumped 64G/128G to 15%; reverted, since that overrode a number Julian had already stated explicitly rather than left open |
| 4 | Compression algorithm (primary + recompression) | Architecture split — **Julian's explicit instruction**: "everything up to a GB goes to zstd primary and done"; lz4-then-recompress is a narrow, uncertain exception "only for the weakest of weak CPU bound machines," not the default for the whole small-tier band | **Built, corrected back to the explicit instruction.** **256M/512M/1G** — `zstd(level=3)` primary, no recompression at all. **2G-128G** — `lz4` primary + `zstd(level=3)` recompression. An earlier version wrongly gave 256M/512M the lz4+recompress architecture, over-applying Julian's narrow "weakest of weak" exception to the whole dire band — a real implementation error, caught directly ("I see you are still disobeying") and reverted. Why the split falls at 1G/2G rather than headroom-based reasoning: **workload compute-boundedness**, per Julian directly. 256M-1G see light, few-user usage with no CPU contention, so paying zstd's cost synchronously is free and grabbing density immediately matters more than waiting ("you need to get whatever you can"). 2G+ increasingly runs compute-bound workloads (LLMs/genAI/many apps) competing for the same CPU a dense primary would consume, so the cheap primary protects that live demand and the expensive pass is deferred to idle time ("you can wait for it later"). Directed, quoted in full in `rationale.md [9]`. Placing the boundary exactly at 1G/2G (rather than Julian specifying every tier) is Claude's own inference. Never step down to level=1. `levels.nix`, `rationale.md [9]`, site pages pending re-sync |
| 5 | Backstop watchdog (last resort) | PSI-based (`memory.pressure` full/avg10 vs threshold, accumulator over the duration window) — confirmed scale-invariant by formula (no RAM/byte term), not just convention; the only genuinely RAM-dependent piece is whether to run the daemon at all (256M's fixed RSS tax vs variable budget), a resource-budget question, not detection-scale | **Built.** 60% flat at every tier — already correct: the real systemd-oomd upstream default, not NixOS's own 80% module default (whose claimed Fedora origin doesn't match Fedora's actual 50%/20s). Duration is 30s shared across every zram tier, but `mode = "zswap"` shortens it to 3s specifically — directed, to react faster under a bursty LLM-compute workload; the 60% threshold itself is unchanged. This is the one place the "zram/zswap share the number" stance didn't hold up once checked against the real fleet data point it's supposed to be grounded in (`modules/oomd.nix`). Protection already complete: `OOMScoreAdjust=-900` + `ManagedOOMPreference=omit` both set (`modules/oomd.nix`). zram/zswap share the threshold, not the duration or the severity behind it — added `oomd.pressureDiagnostics.enable` (`modules/oomd.nix`, `modules/default.nix`), on by default for `mode = "zswap"`: logs `memory.pressure` + `io.pressure` together, diagnostic only, never wired into the kill decision. Full writeup: `rationale.md [10]` (extended) and new `[14]` |
| 6 | Recompression: keep or drop, and when it runs | Long target uptime means the background recompression cost has time to pay off. Cadence should be "whenever there is idle time" (Julian's explicit call), not a fixed calendar — a busy box shouldn't be forced to run it, and a quiet box shouldn't only get one chance a day | **Built.** Off at 256M/512M/1G (zstd-alone primary already dense, row 4); on from 2G up. Cadence: timer checks every 15 min (`recompressionTimer.onCalendar`, now a check-frequency not a run-frequency), gated on CPU PSI (`/proc/pressure/cpu` "some" avg10 < 10%) — acts only when genuinely idle, skips and retries otherwise. `modules/zram.nix`, `rationale.md [11]` |
| 7 | Swappiness | Medium-cost ratio (scale-invariant) is only HALF the story. The other half: how much true RAM remains behind the physical leg, and how much file cache a box has to sacrifice before ever touching anon memory — this genuinely varies by size, and Julian's correction adds it as a second axis | **Built, adversarially revised, boundary corrected, then directly revised twice more.** **120** at 256M/512M/**1G** (eager — unified after Julian confirmed 1G belongs with the dire tiers, not the reluctant ones, matching row 4's compression-architecture correction; value went 180 (Pop!_OS default) → 130 (adversarial revision) → **120**, Julian's own further direct revision). **10 at rest, relief-gated up to 60 under genuine sustained pressure** at 2G-128G (was flat 60 — checked against real precedent for the FLAT SHAPE, which still holds, but 60 itself was never checked against real data; Julian directly questioned it — "the old unraid server had 10, the elitebook has 25" — and 10 is that real historical value, now the resting point, with a new PSI-gated relief valve, row 14, restoring 60's old always-on behavior only during confirmed overflow). zswap: 25, directed — Julian named elitebook directly ("the only zswap box is elitebook"), reversing an earlier reasoned-midpoint value of 120; note this now sits ABOVE zram's new 10-at-rest floor, a genuine unresolved ordering tension exposed by this revision, not patched over (`rationale.md`, zswap profile section). `modules/sysctls.nix`, `modules/zram.nix`, `levels.nix`, `rationale.md [3]`, `[17]` |
| 8 | Watermarks | `boost_factor=0` flat: confirmed, even more clearly justified on small zones (kernel's own bug history — boost is *more* dangerous there). `scale_factor` taper: small-tier boost (200) has real indirect support (`min_free_kbytes`'s sqrt-scaling eats proportionally more of small tiers' headroom); large-tier reduction to 100 has no support found anywhere, and one data point argues the opposite (the mechanism's own motivating case was 140GB *servers* needing MORE headroom) | proposed correction, pending confirmation: keep 200/150 at small/mid tiers, flatten 64G-128G back to 125 (the one sourced anchor) instead of the current 100 — looks like an unsupported, possibly backwards extrapolation |
| 9 | MGLRU `min_ttl_ms` | Kernel doc explicitly calls being "agnostic to... memory sizes" a deliberate design virtue — cleanly confirmed flat, no RAM-size caveat needed | unchanged: 1000ms flat. oomd-interaction flag: no documented conflict found anywhere (the "no oomd" wording is an audience statement, not a warning); one anecdotal complaint exists, no kernel-level evidence — stays genuinely unmeasured, recommend a real tracked experiment alongside 003 rather than closing the question |
| 10 | CPU class as a second dimension | Investigated and rejected as a formal axis. RAM tier doesn't reliably predict CPU headroom (real counterexample: same-tier container on a many-core host vs. a single shared vCPU cloud box) — the instinct is sound. But "shared vs. dedicated" conflates two different properties (per-core speed vs. contention/steal-time), and unlike RAM, CPU class has no stable, checkable ground truth — for a container it can change with zero config edit (noisy neighbor co-scheduled) | **Superseded by row 11** — the actual answer wasn't a CPU-class axis, it was true-RAM-headroom, which nixram already knows per-level without inferring anything. `compressionAlgorithmOverride` still exists as a general escape hatch but is no longer expected to be commonly needed |
| 11 | The master principle: true RAM headroom, not RAM tier or CPU class | Julian's worked examples (e2-micro, vultr, 256M, elitebook, a real 128G-class server), generalized: the deciding variable for swappiness is how much real RAM remains behind the physical leg, and how much file cache exists to sacrifice before touching anon memory. The deciding variable for RECOMPRESSION ARCHITECTURE turned out to be a separate axis — see row 4 | **Built, reviewed nine times, five real corrections landed.** Shape holds: 256M/512M/1G=zstd(level=3)-alone/no-recompress+swappiness 120+ceiling=ram, 2G-128G=lz4+recompress+swappiness 10-at-rest/relief-gated+ceiling=ram*75/100. First pass found real cracks in swappiness (180→130). Second pass caught that *I* had overridden Julian's own explicit 20% figure at 64G/128G with an uninstructed "15%" revision — reverted. Third pass found the "master principle" was wrong for compression architecture specifically (the "1G-island" framing was Claude's own phrase) — left open rather than patched. Fourth pass (asking Julian directly) resolved it: workload compute-boundedness, not headroom (row 4). Fifth pass: Julian caught the resulting 256M/512M architecture ITSELF was wrong — over-applied his narrow "weakest of weak" exception. Reverted to "everything up to a GB goes to zstd primary and done." Sixth pass: Julian confirmed swappiness should unify across 256M-1G too — 1G moved from 60 to the eager value. Seventh pass: Julian caught the ceiling formula (row 13) was incoherent, corrected to his own pi()/3-smooth formula. Eighth pass: Julian gave two further direct corrections in one message — swappiness 130→120, and the ceiling's rounding was wrong at 768M (should be 768, not 512) and ~128G (should be 96, not 64). Ninth pass: Julian questioned whether the reluctant tiers' flat 60 was still too high, citing two more real data points (old Unraid server=10, elitebook=25) — 60 was never actually checked against real fleet data, only the FLAT SHAPE was; resting value dropped to 10, with a new PSI-gated relief valve (row 14) restoring 60 during genuine sustained pressure. One thing still asserted, not data-backed: whether 120 is optimizing the right variable on dire tiers; the relief valve's specific thresholds are also unvalidated |
| 14 | Swap should be for overflow (deploys, bursts) or icecold pages, not routine fullness | Julian directly: "expect the servers to be stuffed to 80% or so and we don't want them to start swapping just because. Swap is for overflow when upgrades run or whatever, or for icecold pages." A static swappiness number, however low, can't distinguish "routine fullness" from "genuine overflow" — it needs a live signal, the same shape as the CPU-PSI idle gate already used for recompression (row 6), applied to memory pressure instead of CPU idleness | **Built.** `zram.swappinessRelief`: a systemd timer (default every 30s) reads `/proc/pressure/memory`'s "some" line. Enters relief (swappiness → 60) when `avg10` >= 10% (fast signal, catches a real spike quickly); leaves relief (swappiness → the tier's 10-at-rest baseline) only when `avg60` < 1% (slower signal, deliberately harder to satisfy, so a brief lull mid-spike doesn't bounce swappiness back down early). On by default only on reluctant tiers (2G-128G) — dire tiers are already eager, nothing to relieve. State tracked under `/run`, cleared every reboot. Unvalidated starting point for every threshold; tune freely. `modules/zram.nix`, `modules/default.nix`, `rationale.md [17]` |
| 12 | The 768M gap: rounding into a discrete policy cliff, not just a percentage — added, then removed | Julian caught that the ladder had no 768M anchor. Checked: every other anchor boundary only shifts continuous percentages (safe under "round up" per `faq.md`). Originally: 1G was the one anchor whose zram architecture differed from both neighbors, so the 513-1023 MiB band silently inherited 1G's policy regardless of real headroom | **Built, then found fully redundant by two later corrections, then removed as part of the full-project prune.** Added `768M` as a 15th anchor, cloning 512M's policy exactly. Two later corrections closed the gaps it existed to bridge: row 4 unified 256M-1G's architecture, and row 7/11 unified their swappiness too. Checked directly: `512M` and `1G` became identical in every formula (30% resident-limit, `ram` ceiling, zstd(level=3)-alone architecture, swappiness=120, watermark=200, oomd=true) — a 768 MiB box rounding up to `1G` gets byte-for-byte the same treatment a dedicated anchor gave it. Rather than leave a fully inert fifteenth anchor around as a documentation footnote, it was removed once the redundancy was confirmed — fourteen anchors again, `256M` through `128G`. `levels.nix`, `checks/default.nix`, `rationale.md [16]` |
| 13 | The zram-size ceiling: derived from the resident-limit budget, not a disconnected formula | Julian caught the ceiling was incoherent ("the ceiling does not make any sense, does it?") — checked against his own four worked examples (256M, 512M/vultr, 1G/e2-micro, ~128G/server), the old three-shape formula (`ram*2` / `ram` / `min(ram/2,16384)`) was ~2x too generous at the small end and 4-5x too SMALL at the large end (a borrowed Pop!_OS 16GiB cap that made no sense once nixram switched to a resident-limit-first safety model) | **Built, refined to Julian's own formula, then simplified to its exact reduction.** Julian's formula: "take the physical ram, multiply by pi and take the nearest base 2ish value" — meaning the nearest 3-smooth number (OEIS A003586: 2 and 3 as the only prime factors — the sizes RAM/VPS actually ship in). Verified as real fasteval built-ins (`pi()`, `log()`, `round()`), but because the resident budget is a FIXED percentage of RAM per tier group, the "nearest 3-smooth number" reduces to the SAME flat fraction for every tier in a group: 30% x pi = 0.9425 → nearest 3-smooth fraction 1.0 (`ram`); 25% x pi and 20% x pi (0.7854, 0.6283) both → 0.75 (`ram * 75 / 100`). No runtime pi()/round() branching needed — two flat percentages produce the identical result. Caught and fixed two intermediate wrong roundings along the way, both against Julian's own direct corrections: 768M was rounding down to 512 MiB (should be 768, its own 3-smooth value: 768 = 1.5x512); ~128G was rounding down to 64 GiB (should be 96, Julian: "96GB is better" — 96 = 1.5x64). Checked against all examples: 256M→256 MiB, 512M→512 MiB, 768M→768 MiB (exact), 1G→1 GiB ("almost a GB"), ~128G→96 GiB (exact, his own correction). `levels.nix`, `rationale.md [1]`, `checks/default.nix`, site pages pending re-sync |

nixram

Declare a RAM level. Everything else follows.

One small NixOS module derives a coherent memory-pressure policy from a single declared RAM level — zram (or zswap) sizing, PSI-armed systemd-oomd, and the vm.* sysctl layer, all from one table. Every value it sets is tagged sourced, extrapolated, or kernel default, so nothing reads as more settled than it is.

    services.nixram.level = "4G";

# then wire the module into your flake + configuration:
# flake.nix
inputs.nixram.url = "github:julian-corbet/nixram-corbet-ch";
# configuration.nix
imports = [ inputs.nixram.nixosModules.nixram ];
services.nixram = {
  enable = true;
  level = "4G";
};

How it works

No eval-time guessing. Nix evaluation is pure and static — it can't read a target machine's live memory — so the RAM level is a fact you detect once and commit, not something the build infers behind your back.

1

Detect once

Run detect-level on the target machine. It reads /proc/meminfo and rounds up to the next of the fourteen anchors, never down, then prints a ready-to-paste line.

2

Paste once

Commit services.nixram.level like any other hardware fact. There is no default level and no eval-time auto-detection — leaving it unset is a hard error, by design.

3

Everything derives

The zram device and its sizing, systemd-oomd's PSI arming, and the whole vm.* sysctl layer all resolve from one table — every value with an escape hatch if you disagree.

The model

nixram's central thesis in one picture: disksize is only a cheap virtual ceiling, and the real physical cost is bounded separately by zram-resident-limit. Budget the physical, and the virtual can afford to be generous.

Virtual ceiling

disksize is cheap address space — deliberately generous where it counts: up to 2× RAM on the smallest tiers and full-RAM through the middle, before the /2 taper and 16 GiB cap bring it back inside upstream's 0.1–0.5 guidance on the largest boxes. Once a resident limit is doing the real safety job, a bigger virtual ceiling costs nothing but lets compression stretch the same physical spend further before hitting a wall.

Physical budget

zram-resident-limit caps the real physical spend inside a conservative fraction of RAM at every tier up to 32G — half of RAM on the smallest boxes, tapering to 35% once there is slack to share. From 64G up it is deliberately unset, because disksize itself already caps physical spend at ≤25% of RAM.

PSI, not percentages

systemd-oomd is armed on pressure-stall time, which is medium-agnostic. A swap-used percentage measured against the inflated disksize denominator is the wrong number, so SwapUsedLimit is not configured at all — not as a decorative backstop, not anywhere.

The levels

Fourteen RAM-size anchors, 256M through 128G, each a complete and coherent set of values. ram is zram-generator's own variable for total detected RAM in MiB — the formulas are evaluated against real /proc/meminfo at boot, so the formula is what's tiered, not a number baked in per level.

Level by level — why each tier is shaped the way it is

256M — the survival tier

Everything here is rationed twice over: RAM, and the CPU to compress it. disksize is ram * 2 because compression is the only headroom this box will ever have, and the resident limit at half of RAM caps how much of that reach can become physical spend. zstd(level=1) rather than plain zstd: every page fault costs CPU on a machine that has almost none to spare. systemd-oomd stays off — its own RSS is an unmeasured fraction of a very small total — so the kernel OOM killer plus the OOMScoreAdjust = -900 protection layer stands guard alone. Recompression off: a pool that tops out around ~512M logical can't repay a second pass.

512M — first full-strength tier

The same 2× / ½ shape as 256M, but nothing needs rationing twice anymore: plain zstd is affordable, oomd is armed (its RSS is a far smaller fraction of the box), and the daily idle-recompression pass starts paying — tens of MB back, on a machine where tens of MB are percent-of-RAM money.

1G — the small-box shape holds

The last tier of the small-box shape: double-RAM virtual reach, half-of-RAM physical budget. From here down the box lives or dies by compression stretch. watermark_scale_factor stays at 200 across all three small tiers: any fixed percentage of a tiny zone is a tiny absolute number of free pages, so kswapd must start reclaiming earlier to leave real headroom.

2G — the sourced middle begins

disksize drops to plain ram — Fedora's own shipped default (full-RAM scaling), the best-sourced point on the entire curve. The physical budget tapers from ½ to 35%: with multiple GB present, the compressed pool no longer needs to be allowed half the machine to be useful. Watermarks relax to 150.

4G — the reference tier

The most ordinary row in the table — disksize sourced from Fedora, the standard 35% budget, every shared constant at its normal value. To understand nixram's model, read this row first: every other tier is this row with one scarcity turned up or down.

6G / 8G — same row, more slack

Nothing changes but the numbers the same expressions resolve to. 8G is the last tier where a full-RAM disksize is still the right ceiling — it's also exactly where Fedora's own default stops scaling (min(ram, 8192)). Past 8 GiB, a swap absorber sized to the whole machine is more than a box that large can usefully drain before a timely OOM decision beats stretching compression any further, so at 10G the ceiling hands over to the bounded /2 taper.

10G — the taper begins

disksize switches to min(ram / 2, 16384). The need for swap capacity stops scaling with RAM: a 10G box under pressure doesn't want 10G of swap, it wants a bounded absorber and then a timely OOM decision. The /2 taper is what binds through 32G (5G of reach here); the 16 GiB cap in the formula is Pop!_OS's shipped ceiling, waiting to take over. Watermarks reach the Pop-validated 125.

12G / 16G / 24G — the plateau

The /2 taper walks the ratio down: 6G of reach on a 12G box, 8G on 16G, 12G on 24G. The shrinking fraction is the point, not a side effect — what absorbs a bounded working-set overflow is an absolute number of GiB of cold-page parking, not a fixed percentage of a machine that keeps growing. Once the absorber is big enough to hold the cold set, more capacity only delays the OOM decision a larger box should be making sooner, so the reach is deliberately let fall as a share of RAM while staying inside the same 35% physical budget.

32G — where taper meets cap

32 / 2 = 16: the one tier where the taper and the 16 GiB cap agree exactly. Above this line the cap alone rules and RAM size stops mattering to the absorber entirely.

64G — the budget retires

The resident limit is deliberately unset. disksize (16G) is already ≤25% of RAM — a physical bound tighter than the 35% rule would produce — so a second cap would be decoration, not protection. Flagged as an open question rather than proven. Watermarks ease to 100: kswapd can afford to be lazy here. For a box running one huge, non-swap-shaped workload, the honest alternative is mode = "none" — oomd and sysctls without any swap medium.

128G — the cap is the whole story

16G of absorber on a machine with eight times that much RAM. zram at this scale is not survival — it's a parking lot for cold pages that would otherwise squat on hot RAM for months of uptime. Everything else as 64G.

Decisions

The deliberate calls — the places nixram departs from a default on purpose, and says so out loud. Each links to the numbered reasoning behind it.

deliberate departure

Exceeds upstream guidance — deliberately

zram-generator recommends a disksize fraction of 0.1–0.5 of RAM. The small and mid tiers exceed it — up to 2× RAM below 2G, full-RAM at 2G–8G — while the 10G+ /2 taper and 16 GiB cap bring disksize back to ≤0.5× RAM (0.25× at 64G, 0.125× at 128G). Exceeding it where it does isn't an oversight: disksize is only the virtual ceiling, and zram-resident-limit does the real physical-safety job, so a generous ceiling just gives compression room to stretch.

SwapUsedLimit: not configured anywhere

A swap-used percentage is measured against disksize — which nixram deliberately inflates past the real budget. It would read "plenty of headroom" right up until the resident-limit wall is hit. Not a decorative backstop, not an oversight — it is not configured at all.

PSI-only OOM detection

systemd-oomd is armed on memory-pressure stall time — 60% over 30s, systemd's own upstream defaults — and nothing else. Stall time is medium-agnostic: it doesn't care whether swap is zram, zswap, or a disk partition, or how large its nominal capacity looks.

Detect once, paste once

Nix evaluation is pure and static — it can't read a machine's live /proc/meminfo. A silent eval-time guess would trade a wrong policy for the appearance of convenience, so level has no default and leaving it unset is a hard error. You run the detector once and commit the line.

Read the FAQ →

Idle recompression, two-phase

The kernel never recompresses on its own; userspace drives it. A rolling timer marks the resident set idle on one firing and recompresses what stayed untouched on the next, so "idle" reflects a real dwell period. The dense zstd(level=12) idle tier is free at read time — zstd's decompression speed is essentially level-independent.

An honesty taxonomy

Every value carries a tag: sourced, extrapolated, or kernel default. Extrapolated values are reasoned, not measured — open experiments, not settled facts — and results feed back into the table as tag upgrades over time.

Get started

Two moves: find your level on the target machine, then wire the module into your flake.

Add the flake

Point an input at the flake, import the module, and enable it at your level.

# flake.nix
inputs.nixram.url = "github:julian-corbet/nixram-corbet-ch";

# in your nixosSystem modules:
imports = [ inputs.nixram.nixosModules.nixram ];
services.nixram = {
  enable = true;
  level = "4G";
};

Find your level

Run this on the target machine — it prints the exact line to paste above.

nix run \
  github:julian-corbet/nixram-corbet-ch#detect-level

No default level, no auto-detection — by design. Run the detector on the target machine once, then commit the line it prints like any other hardware fact. A build can't read a machine's live memory, and a silent guess would be a footgun, not a convenience.

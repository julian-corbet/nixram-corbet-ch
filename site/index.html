a homage to nix + RAM — colloquially, you don't have RAM

nixram

Declare a RAM level. Get coherent zram/zswap sizing, PSI-armed systemd-oomd, and vm.* sysctl tuning — derived from one table, instead of a dozen loosely-related knobs you hand-pick and hope agree with each other.

GitHub repository → nix run github:julian-corbet/nixram-corbet-ch#detect-level

01The thesis

zram's disksize is only a cheap, virtual ceiling, and the real physical cost is bounded separately by zram-resident-limit, kept inside a conservative fraction of RAM at every tier — so disksize can afford to be generous, and compression gets room to stretch the same physical spend further before hitting a wall. systemd-oomd is armed on PSI (pressure stall information), not on swap-used percentage, because stall time is medium-agnostic and a percentage-of-disksize detector would be reading the wrong number under this sizing model. Every value nixram sets is tagged sourced, extrapolated, or kernel default, so nothing here is presented as more settled than it actually is.

02The instrument

Pages fall out of ordinary memory and land in the compressed pool. Watch what happens to their size on the way in.

A diagram: small squares representing memory pages drift downward and shrink as they compress into a pool at the bottom of the frame. The pool's physical height is capped by a solid amber line labeled "zram-resident-limit — the real budget". A taller dashed teal line further up is labeled "disksize — the virtual ceiling". Because pages shrink as they compress, more logical pages fit under the amber line than would fit if they stayed full size.

Amber — zram-resident-limit, the real, physical wall. Teal, dashed — disksize, a generous virtual ceiling that costs nothing once the amber line is doing the actual safety job.

04Decision cards

Every card is one decision, one line of reasoning, and its honesty class. Two are flagged: one because it deliberately breaks with upstream guidance, one because it deliberately configures nothing at all.

Amber-flagged

Disksize exceeds upstream guidance — deliberately

zram-generator's own docs recommend sizing disksize to 0.1–0.5 of RAM. Every level here exceeds that, up to 2x RAM at the smallest tiers, because disksize is only the virtual ceiling once a resident limit is doing the real physical-safety job.

◐ extrapolated curve, sourced endpoints rationale.md [1] →
Red-flagged

SwapUsedLimit / ManagedOOMSwap: not configured anywhere

Not a decorative backstop left at a safe default — genuinely absent, at every level. A swap-used percentage reads the wrong number once disksize is deliberately inflated beyond the real budget; PSI has no such blind spot.

dropped by decision, not by oversight faq.md →

PSI-only OOM detection

systemd-oomd is armed on memory-pressure stall time, never on swap-used percentage. Stall time is medium-agnostic — it doesn't care whether the swap is zram, zswap, or disk.

● sourced (systemd-oomd upstream defaults) rationale.md [10] →

Detect once, paste once

Nix evaluation is pure and static — it can't read a target machine's live /proc/meminfo. No default level, no eval-time auto-detection: a detector app prints a line you paste into your own config and commit, like any other hardware fact.

a deliberate non-goal, not a missing convenience faq.md →

Idle recompression: two-phase rolling timer

A naive mark-then-recompress pass would recompress the entire pool every run. Marking and recompressing are split across two timer firings, so "idle" means one full interval of real non-use. Idle tier: zstd(level=12) — the densest setting below zstd's optimal-parser cost cliff, not 19.

◐ extrapolated design, sourced kernel primitive rationale.md [11] →

No min_free_kbytes formula

No distro default or kernel doc reviewed offers a universal per-tier formula that generalizes across 256M–128G. Rather than invent one, the kernel's own computed value is left untouched at every level — an override exists only for operators who've measured their own workload.

○ kernel default — not a nixram opinion rationale.md [6] →

The honesty taxonomy itself, as a feature

Every tunable is tagged sourced, extrapolated, or kernel default — out loud, in the code comments, the table, and here. Nothing is presented as more settled than it actually is, and the numbering behind each tag is load-bearing across the whole module.

the taxonomy is the product, not documentation of it rationale.md →

05Quickstart

{
  inputs.nixram.url = "github:julian-corbet/nixram-corbet-ch";
}
# in your nixosSystem modules:
imports = [ inputs.nixram.nixosModules.nixram ];
services.nixram = {
  enable = true;
  level = "4G";  # find yours: nix run github:julian-corbet/nixram-corbet-ch#detect-level
};
